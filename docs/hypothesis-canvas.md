# ユビキタス言語システム - 仮説キャンバス

## 1. 私たちは信じています…

**ドメイン駆動設計（DDD）を実践するチームが、ユビキタス言語を効果的に管理・共有・進化させるための専用システムを構築することで、ドメインの理解を深め、コミュニケーションの質を向上させ、より良いソフトウェア設計を実現できる。**

### 背景と前提

- DDDにおけるユビキタス言語は、開発チームとドメインエキスパートが共通の語彙を使用するための重要な概念
- 現状、ユビキタス言語は散在的に管理されており（Wiki、ドキュメント、コード、会話など）、一元管理されていない
- 言語の定義が曖昧なまま開発が進むと、認識のズレや実装のミスマッチが発生する
- ドメインの理解が深まるにつれて、ユビキタス言語も進化・変更される必要がある

### 解決したい課題

#### 顕在課題
- ユビキタス言語の用語が複数のドキュメントに散在し、最新の定義を見つけにくい
- 同じ概念に対して異なる用語が使われ、混乱が生じる
- 新規参加メンバーがドメイン用語を理解するのに時間がかかる
- コードとドキュメントで用語の不一致が発生する

#### 潜在課題
- ドメインの深い理解が個人の頭の中に留まり、チーム全体で共有されない
- 境界づけられたコンテキスト間での用語の使い分けが明確でない
- 言語の変更履歴が追跡できず、なぜその用語が選ばれたのか背景がわからない
- ドメインエキスパートと開発者の間で、言語の理解に乖離がある

## 2. 誰のために…

### プライマリターゲット

1. **DDDを実践する開発チーム**
   - アジャイル/スクラム開発チーム
   - モデリングを重視するチーム
   - 複雑なドメインを扱うチーム

2. **ドメインエキスパート/ビジネスアナリスト**
   - 業務知識を持つステークホルダー
   - ビジネスルールを定義する人々
   - 要件を整理・伝達する役割

### セカンダリターゲット

3. **プロダクトオーナー/プロジェクトマネージャー**
   - チーム間のコミュニケーションを促進する役割
   - プロダクトビジョンを伝える役割

4. **新規参加メンバー/オンボーディング対象者**
   - プロジェクトに新しく参加した開発者
   - ドメインについて学ぶ必要がある人々

### ペルソナ例

**開発者 - 田中太郎（28歳）**
- 経験3年のバックエンドエンジニア
- DDDの基礎知識はあるが、実践経験は少ない
- 新しいプロジェクトに参加し、ドメイン用語を素早く理解したい
- ドキュメントが散在していて、どれが最新か判断できず困っている

**ドメインエキスパート - 佐藤花子（42歳）**
- 業界経験15年のビジネスアナリスト
- 業務知識は豊富だが、技術的な詳細には詳しくない
- 開発チームに業務ルールを正確に伝えたいが、技術用語との対応がわかりにくい
- 自分の説明が正しくコードに反映されているか確認したい

## 3. 達成するために…

### 期待される成果（定量的）

1. **学習時間の短縮**
   - 新規メンバーのドメイン理解時間を **50%削減**
   - オンボーディング期間を **2週間から1週間に短縮**

2. **コミュニケーション品質の向上**
   - 用語の認識齟齬によるバグレポートを **40%削減**
   - ドメインエキスパートとの会議時間での認識合わせ時間を **30%削減**

3. **ドキュメント品質の向上**
   - ユビキタス言語の用語カバレッジを **80%以上に向上**
   - コードとドキュメントの用語一致率を **90%以上に向上**

4. **チーム生産性の向上**
   - 用語確認のための中断を **週5回から週1回に削減**
   - モデリングセッションの効率を **25%向上**

### 期待される成果（定性的）

1. **ドメイン理解の深化**
   - チーム全体でドメインの深い理解を共有
   - ドメインエキスパートと開発者の認識ギャップを解消

2. **設計品質の向上**
   - ユビキタス言語に基づいた一貫性のある設計
   - 境界づけられたコンテキストの明確化

3. **知識の蓄積と継承**
   - ドメイン知識の組織資産化
   - 言語進化の過程と理由の記録

## 4. これをどのように測定しますか？

### 定量的指標（KPI）

#### オンボーディング関連
- **新規メンバーオンボーディング時間**
  - 測定方法: 参加から初めて有意義なコントリビューションまでの日数
  - 目標: 14日 → 7日
  - 測定頻度: 新規メンバー参加時

- **ドメイン用語理解度テスト**
  - 測定方法: 主要30用語の理解度チェックリスト正答率
  - 目標: 1週間後に80%以上
  - 測定頻度: オンボーディング1週間後

#### コミュニケーション関連
- **用語齟齬によるバグ/課題数**
  - 測定方法: Issue/バグトラッカーでの「用語理解の誤り」タグ件数
  - 現状ベースライン: 月10件
  - 目標: 月6件以下
  - 測定頻度: 月次

- **ドメインエキスパートとの会議での確認時間**
  - 測定方法: 会議時間のうち「用語の意味確認」に使われた時間の割合
  - 現状ベースライン: 30%
  - 目標: 15%以下
  - 測定頻度: 四半期ごとのレトロスペクティブで回顧

#### ドキュメント品質関連
- **ユビキタス言語カバレッジ**
  - 測定方法: システムで管理されている用語数 / ドメインの主要概念総数
  - 目標: 80%以上
  - 測定頻度: 月次

- **コード・ドキュメント用語一致率**
  - 測定方法: 静的解析でクラス名/メソッド名とユビキタス言語の一致度
  - 目標: 90%以上
  - 測定頻度: CI/CDパイプラインで自動測定

- **用語定義の最新性**
  - 測定方法: 過去3ヶ月以内にレビューされた用語の割合
  - 目標: 100%
  - 測定頻度: 月次

#### 利用状況関連
- **システム利用頻度**
  - 測定方法: 週あたりのアクティブユーザー数、検索回数
  - 目標: チームメンバーの80%が週1回以上利用
  - 測定頻度: 週次

- **用語参照・検索頻度**
  - 測定方法: 各用語の閲覧回数
  - 目標: 主要用語は週5回以上参照
  - 測定頻度: 週次

### 定性的指標

#### ユーザーフィードバック
- **チームサーベイ**
  - 測定方法: 四半期ごとのチームアンケート（5段階評価）
  - 質問項目:
    - 「ドメイン用語の理解度」
    - 「システムの有用性」
    - 「コミュニケーションの改善」
  - 目標: 平均4.0以上
  - 測定頻度: 四半期ごと

- **ドメインエキスパート満足度**
  - 測定方法: インタビュー形式のフィードバック
  - 確認項目:
    - 「自分の意図が正しく開発チームに伝わっていると感じるか」
    - 「システムを使うことで説明が楽になったか」
  - 測定頻度: 半期ごと

#### 行動変化の観察
- **モデリングセッションの質**
  - 測定方法: ファシリテーターの観察記録
  - 観察項目:
    - 用語の定義で議論が停滞する頻度
    - 参加者の共通理解の速度
    - 新しい洞察が生まれる頻度
  - 測定頻度: セッション後の振り返り

- **コードレビューでの指摘**
  - 測定方法: プルリクエストでの「用語の使い方」に関するコメント数
  - 期待: コメント数の減少（理解が共有されている証拠）
  - 測定頻度: 月次集計

### 測定データの収集方法

1. **自動収集**
   - システム内アクティビティログ
   - 静的解析ツールとの連携
   - CI/CDパイプラインでの自動チェック

2. **手動収集**
   - 月次レビューミーティング
   - 四半期レトロスペクティブ
   - 新規メンバーオンボーディングチェックリスト

3. **継続的改善**
   - 指標の定期的な見直し（四半期ごと）
   - 目標値の調整
   - 新しい指標の追加

## 5. 影響、肯定的または否定的

### 肯定的な影響

#### チームレベル
- **コミュニケーションの円滑化**
  - 共通言語による認識のズレ削減
  - 議論の焦点が明確になり、建設的な対話が増える

- **設計品質の向上**
  - ドメインに忠実な設計が可能になる
  - リファクタリングの方向性が明確になる

- **知識の民主化**
  - ドメイン知識が特定の人に集中しなくなる
  - チーム全体のスキルレベルが底上げされる

#### 組織レベル
- **ドメイン知識の資産化**
  - 組織の知的資産として蓄積
  - 人の入れ替わりに対する耐性向上

- **クロスチーム連携の改善**
  - 異なる境界づけられたコンテキスト間の関係性が明確になる
  - チーム間の用語マッピングが容易になる

#### プロダクトレベル
- **要件とコードの一致**
  - ビジネスルールが正確にコードに反映される
  - トレーサビリティの向上

- **保守性の向上**
  - コードが読みやすくなる
  - 変更時の影響範囲が理解しやすくなる

### 否定的な影響（リスク）

#### システム導入のオーバーヘッド
- **学習コスト**
  - リスク: チームメンバーが新しいシステムの使い方を学ぶ必要がある
  - 対策: シンプルなUX設計、段階的な導入、トレーニングセッション

- **メンテナンスコスト**
  - リスク: 用語定義の更新・レビューに時間がかかる可能性
  - 対策: 更新プロセスの自動化、定期的なレビュー時間の確保

#### 過度な形式化のリスク
- **形式主義の罠**
  - リスク: 用語の定義や管理が目的化し、本質的なドメイン理解がおろそかになる
  - 対策: 軽量なプロセス設計、必要最小限のルール、実用性重視

- **柔軟性の低下**
  - リスク: 厳密すぎる用語管理により、探索的な議論が阻害される
  - 対策: 「暫定的な定義」の許容、段階的な精緻化のサポート

#### 技術的リスク
- **システム依存**
  - リスク: システムが使えない場合に業務が止まる
  - 対策: エクスポート機能、バックアップ、オフライン閲覧

- **スケーラビリティ**
  - リスク: 用語数が増えた場合の検索性能低下
  - 対策: 効率的な検索機能、階層構造、タグ付け

#### 組織的リスク
- **導入抵抗**
  - リスク: 既存のやり方に慣れたメンバーからの抵抗
  - 対策: 早期からの巻き込み、価値の可視化、パイロットプロジェクト

- **不統一な運用**
  - リスク: チームによって使い方がバラバラになる
  - 対策: ガイドラインの整備、成功事例の共有、コミュニティ形成

### リスク軽減戦略

1. **段階的導入**
   - Phase 1: 小規模チームでのパイロット運用
   - Phase 2: フィードバックを反映した改善
   - Phase 3: 組織全体への展開

2. **継続的改善**
   - 定期的なレトロスペクティブ
   - ユーザーフィードバックの収集と反映
   - 柔軟なプロセス調整

3. **コミュニティ形成**
   - ベストプラクティスの共有
   - 疑問点を解決する場の提供
   - チャンピオンの育成
